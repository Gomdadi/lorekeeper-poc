# 비교 ③ 프롬프트 보강 후 medium vs high (run3)

> "추출 관점 체크리스트 + Event 입도 + RELATED_TO 적극 + 고립 노드 금지 + 중복 방지"를 넣은 뒤 실행.
> 원본: `luna_medium_run3.md`, `luna_high_run3.md`. **결론: 보강이 역효과 — 과추출·깨진 관계 증가.**

## 토큰·비용
| | 입력 | 출력 | 비용 |
|---|---|---|---|
| medium | 26,323 | 3,952 | $0.0500 |
| high | 26,410 | 15,936 | $0.1220 |
(입력이 ~25K→~26.3K로 소폭 증가 = 프롬프트 길어진 값, 비용 영향은 미미)

## 그래프 (run3)
| 항목 | medium run3 | high run3 |
|---|---|---|
| Event | 8 | 6 |
| Item | 1 (멸살법) | **4** (멸살법, **세카이**, **멸살법 추천글**, 자전거) |
| 선물(첨부파일) | ✗ | ✗ **(이전 high는 잘 잡던 걸 놓침)** |
| 자전거 | ✗ | ✓ 소유=상실 |
| CharacterState | 2 (신분만, **소속 전무**) | 4 |
| 소속 | ✗ 없음 | 유상아→인사팀 ✓ |
| RELATED_TO | **`김독자-[동료]->김독자` (자기참조 버그)** | **`김독자-[None]->유상아` (type 누락)** |
| INVOLVED_WITH | 정상 | **`김독자-[None]->세카이` (role 누락)**, `김독자-[저자]->멸살법 추천글`(게시글을 작품으로 과추출) |

## 새로 생긴 문제 (보강 부작용)
- **과추출**: high가 `멸살법 추천글`(게시판 글)을 Item으로, `세카이`(개그성)를 다시 Item으로 — "고립 노드 금지 + 관계 적극 포착 + 의미 있는 것 다 잡기"가 **경계 사례까지 끌어올림**.
- **깨진 관계**: `RELATED_TO type=None`, `INVOLVED_WITH role=None`, `동료 자기참조(김독자→김독자)` — "암시된 관계 적극 포착"이 근거 약한 관계를 억지로 만들며 속성을 못 채움.
- **핵심 누락**: high가 이전엔 안정적으로 잡던 **선물(첨부파일)을 놓침**. 노이즈(추천글·세카이)에 밀림.
- **medium 소속 전무**: "변하지 않은 상태 중복 금지"가 과하게 작동해 **소속을 아예 안 만듦**(중복 방지가 생성 억제로 번짐).

## 결론
- **보강 전(run1/run2)이 더 나았다.** 특히 high는 보강 전 선물을 두 run 다 잡았는데, 보강 후엔 선물을 놓치고 추천글·세카이 같은 노이즈를 만들었다.
- 원인 진단: 입력비가 싸다고 프롬프트를 늘렸지만, **길이·"적극 포착" 톤이 오히려 (a)핵심 규칙을 희석하고 (b)경계 사례 과추출·억지 관계를 유발**. reasoning 모델엔 "더 자세히·더 적극적으로"가 정확도로 직결되지 않음.
- **권고: 이번 보강(추출 관점 체크리스트·Event 입도·RELATED_TO 적극·고립 금지·중복 방지)을 되돌리거나 대폭 축소.** 특히 "고립 금지 + 관계 적극"이 과추출을 부른 주범.
