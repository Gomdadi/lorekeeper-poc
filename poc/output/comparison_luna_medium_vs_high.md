# luna medium vs high 비교 분석 (ch1~2)

> 모델 `gpt-5.6-luna` 고정. medium은 **기존 DB 결과**(재실행 안 함), high는 **신규 실행**.
> 원본 덤프: `luna_medium.md`, `luna_high.md`. 단가(per 1M): 입력 $1.00, 출력 $6.00.
> ⚠️ 각 조건 1회 표본이라 run 편차가 섞여 있음(특히 medium 토큰은 이 run 미기록 → 직전 동일 설정 run 실측 참고치).

## 1. 토큰·비용

| | 입력 토큰 | **출력 토큰** | 합 | **비용** |
|---|---|---|---|---|
| medium | ≈24,797 | ≈4,275 | ≈29,072 | **≈$0.0504** |
| high | 25,087 | **15,703** | 40,790 | **$0.1193** |
| 배수(high/med) | ~1.0x | **~3.7x** | ~1.4x | **~2.4x** |

- 입력은 거의 동일(프롬프트 같음). 비용 차이는 **출력(추론) 토큰**이 만든다 — high가 출력 3.7배.
- high는 medium의 **약 2.4배 비용**. (참고: 이전 실험의 gpt-5.4-mini high는 $0.237로, luna high는 그 절반 수준.)

## 2. 그래프 비교

| 항목 | medium (기존) | high (신규) |
|---|---|---|
| Event | 10 | 9 |
| Item | 2 | **3** |
| **자전거(소지품 상실)** | ✗ 누락 | ✅ `유상아 소유=상실 -ABOUT-> 유상아의 자전거` |
| **선물(첨부파일)** | ⚠ Item만 있고 **소유·관계 없이 고립** | ✅ `김독자 소유=보유 -ABOUT-> 첨부 파일` (완전) |
| CharacterState | 6 (**중복 있음**) | 6 (**중복 없음**) |
| 김독자 소속/신분 중복 | ✗ 소속×2, 신분×2 (ch1·ch2 중복) | ✅ 각 1개 |
| 유상아 소속 | ✅ 인사팀 (Org 2) | ⚠ 대기업 계열사 (부서 뭉갬, Org 1) |
| 동료 관계 | ✅ `김독자-[동료]->유상아` | ✗ 없음(이 run) |
| story_order 과거 승진 | 2.1(현재 취급) | ✅ 1.9(과거 정확) |
| 세카이/싱샹숑 배제 | ✅ | ✅ |
| 일시적 상태(왼팔 통증) | ✅ 없음 | ✅ 없음 |

## 3. 분석

**high가 확실히 이기는 지점 (구조 완전성)**
- **자전거·선물 둘 다 소유 상태 + ABOUT까지 완전 추출**. medium은 자전거를 통째로 놓치고, 선물은 Item만 만들고 소유·관계를 안 이어 고립 노드로 남겼다 — 바로 이번에 사용자가 지적한 약점.
- **CharacterState 중복 없음**. medium은 ch1·ch2가 같은 배경 상태(김독자 소속·신분)를 각각 뽑아 중복 노드가 생겼는데, high는 배경 컨텍스트를 참고해 중복을 억제.
- **과거 사건(승진) story_order 1.9로 정확 배치**.

**medium이 이기는 지점**
- **비용 ~40%**(약 2.4배 저렴).
- **부서 입도**: 유상아 소속을 인사팀으로(high는 상위 회사로 뭉갬). Organization도 2개 vs 1개.
- 동료 관계를 이 run에서 잡음(high는 놓침).

**공통**
- 세카이(개그성 작품) 배제, 일시적 통증 배제 모두 정상.

## 4. 결론

- **구조 완전성(소지품 소유·중복 억제·시간순)에서는 high가 명확히 우수** — 특히 자전거·선물의 소유 상태를 제대로 잡는 건 high뿐이었다. 추론을 키우면 "Item을 만들었으면 소유·관계까지 잇는다" 같은 **다단계 구조 완성**이 잘 된다.
- **대신 비용 ~2.4배**, 그리고 부서 입도·일부 관계는 오히려 medium이 나은 부분도 있어 **완승은 아니다**.
- **표본 1회씩이라 변동 주의** — high의 "자전거·선물 완전 추출"과 medium의 "선물 고립·자전거 누락"이 매번 그대로 재현된다는 보장은 없다. 확정하려면 각 3~5회 반복 집계 필요.

**권고**: 소지품 소유·구조 완전성이 중요한 충돌 탐지 목적에는 **luna high가 비용 대비 값어치**(자전거·선물 같은 검증 대상 상태를 놓치지 않음). 비용이 민감하면 medium을 쓰되, "Item 생성 시 소유·관계 필수" 프롬프트 보강 + CharacterState 중복 dedup 후처리를 붙여 medium의 두 약점을 메우는 방향을 검토.
