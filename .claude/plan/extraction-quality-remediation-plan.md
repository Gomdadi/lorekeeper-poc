# 추출 품질 보완 — 감사 발견 4건 대응

## Context

1~6화 그래프-원문 전수 감사에서 오병합 0·환각 0·수치오류 0으로 구조 품질은 확인됐으나,
**추출 서술 품질** 이슈 4건이 남았다(self-loop는 `produceSelfRel:false`로 별도 해결 완료).

| # | 이슈 | 위치 | 심각도 |
| --- | --- | --- | --- |
| 1 | "유상아와 자신의 **생존**을 불안하게 예상" — 원문은 **죽음**을 예상("아마 이 여자는 죽을 것이다. 그리고 나도.", 생존자 둘은 이현성·김남운) | 5화 Event 5.2 description | 확실 |
| 2 | "육군 중위 **이현성이라고 밝히며**" — 원문에서 이현성은 계급·부대만 발화, 이름은 김독자의 내적 인지 | 4화 Event 4.2 description | 경미 |
| 3 | "폭발의 원인은 원문에 **밝혀지지 않는다**" — 원문은 '녀석이 움직였구나'로 원인을 암시 | 5화 Event 5.4 description | 판단여지 |
| 4 | 김독자 전용 스킬 **'등장인물 일람' 누락** — 발동 메시지 명시 + 김남운 정보창 12개 State의 출처 스킬. 같은 화의 '전지적 독자 시점'은 잡혀 비대칭 | 6화 CharacterState | 실질 |
| 5 | **'성좌' 집단 미추출** — 3화 채널 입장, 6화 김남운에게 관심. 시나리오·후원 메커니즘의 핵심 세력인데 노드 없음(사용자 결정: Organization으로 포함) | 3화~ Organization | 실질 |

**근본 원인 분석** (각 이슈가 재발 가능한 '유형'인지가 대응 수단을 가른다):

- **#2가 가장 구조적이다.** 이 작품의 전제가 '주인공(김독자)이 발화되지 않은 정보를 원작 지식으로
  안다'이므로, **발화 vs 내적 인지의 귀속 구분**은 앞으로 회차마다 반복될 위험 유형이다 → 규칙화 가치 높음.
- #1은 예상·판단을 요약하며 방향(생존/죽음)을 완곡하게 **반전**한 이해 오류 → 방향 보존 규칙으로 억제.
- #3은 few-shot 예시2가 가르친 '한계 명시' 도피처 패턴의 **과적용**(단서가 있는데 "없다"고 단정) → 적용 조건 정밀화.
- #4는 "명시 제시된 상태는 빠짐없이" 규칙이 있음에도, **발동 메시지를 일시 이벤트로 오독**해 보유 상태로
  안 만든 케이스 → 발동/획득 메시지 = 보유 명시임을 못박음.
- #5는 개별 이름 없는 집합 세력('성좌들')을 **관중·배경으로 오판**해 걸러낸 케이스. few-shot 예시1의
  '반란군'(무명 구성원 세력)과 같은 유형이나, "조직"이라는 어감에 안 걸렸다 → 집합 세력 규칙으로 명시.
  단, 이 작품 특성상 이후 개별 성좌가 이름으로 등장하므로 **집합=Organization / 개별=Character** 구분을
  함께 규칙화한다(집단 노드로 뭉개지는 것 방지).

**대응 전략**: 프롬프트 가드레일 4건 보강 → (이미 예정된 PART_OF 재인덱싱에 편승해) 1~6화 재인덱싱 →
표적 검증 → **재발분만 Cypher 수동 패치**. LLM 비결정성 때문에 규칙 보강만으로 100% 보장은 없으므로
수동 패치를 수렴 장치로 둔다.

## 변경

수정 대상 **2개 파일**(`extractor.py`, `extraction_examples.py`). `schema.py`는 무변경 —
이번 보강은 구조가 아니라 서술 습관 교정이라 프롬프트 레이어가 적절하고, 추출 *대상*을 바꾸지 않아야
회귀 기준선이 유지된다(kg-redesign 검증(6)과 같은 논리).

**작품 중립 원칙**: 프롬프트·스키마·few-shot에는 특정 작품의 고유명사·전용 개념을 넣지 않는다.
모든 규칙은 장르 중립(또는 조건부 — 해당 형식이 없는 작품에선 자동 무시)으로 쓴다. 작품 고유
인스턴스(성좌·이현성 등)는 **검증·수동 패치에만** 등장한다 — 이는 이 코퍼스에서 규칙이 작동했는지
확인하는 대상이지 규칙이 아니다.

### 1. `poc/src/extractor.py` — 도메인 규칙 보강 3곳

**(a) 발화 vs 내적 인지 구분 — 신규 불릿** (#2 대응, RELATED_TO 불릿 근처에 추가):
```
- 서술 귀속: '인물이 직접 말하거나 드러낸 것'과 '다른 인물이 속으로 알아본 것'을 구분해 쓴다.
  원문에서 발화되지 않은 이름·정체·사실을 그 인물이 스스로 밝힌 것처럼 쓰지 않는다
  (예: 계급만 밝힌 인물의 이름을 '~라고 밝혔다'로 쓰지 말 것 — 이름은 알아본 쪽의 인지로 서술).
```

**(b) 예상·판단의 방향 보존** (#1 대응, description 규칙 불릿에 한 문장 추가):
```
인물의 예상·판단·감정을 요약할 때 그 방향(긍정/부정, 생존/죽음, 성공/실패)을 원문 그대로
유지한다 — 완곡하게 바꾸다 방향이 뒤집히면 사실 왜곡이다.
```

**(c) 발동·획득 표기 = 보유 상태** (#4 대응, '명시 제시' 불릿에 한 문장 추가. 형식 중립 —
상태창·시스템 메시지가 없는 장르에서는 조건이 성립하지 않아 자동 무시됨):
```
능력·스킬·특성·칭호·업적의 발동/획득/달성이 형식화된 표기(시스템 메시지·상태창·공지 등)로
명시되면 각각 그 인물의 상태로 빠짐없이 만든다 — 발동·획득 표기는 보유의 명시다.
```

**(e) 집합 세력 추출** (#5 대응, Organization 불릿에 추가. 장르 중립 문구):
```
개별 이름 없이 집합적으로만 언급되는 세력·집단(관중 세력·후원 세력·군중 등)도, 단순 배경이
아니라 사건의 진행이나 인물의 행동·운명에 실제로 개입하면 Organization으로 만든다(구성원이
무명이어도 세력 자체가 서사에 개입하면 배경이 아니다). 그 집단의 개별 구성원이 이름·호칭을 갖고
행위 주체로 등장하면 별도 Character로 만든다(집단 Organization 노드로 뭉개지 않는다).
```

### 2. `poc/src/extraction_examples.py` — 한계 명시 조건 정밀화 1곳

**(d)** (#3 대응) 예시2 주의 2("…한계를 명시할 뿐 지어내지 않는다") 문장에 조건 추가:
```
단, 원문이 단서·암시를 주는 경우에는 '~로 암시된다'처럼 암시 수준으로 쓰고,

```

> 불릿들은 건드리지 않는다**(범위 변화 = 회귀 기준선 붕괴).

### 3. 수동 패치 (재인덱싱 후 재발분만, Cypher)

- #1·#2·#3 재발 시: 해당 Event의 `description`을 `SET`으로 교정(원문 근거 문구로).
- #4 재발 시: CharacterState 생성 + 배선:
  ```cypher
  MATCH (c:Character {name:'김독자'}), (e:Event {chapter:6, story_order: <해당 사건>})
  CREATE (s:CharacterState {name:'전용 스킬 등장인물 일람 보유',
          description:'전용 스킬 등장인물 일람이 발동해 김남운의 인물 정보창이 표시되었다.'})
  CREATE (c)-[:HAS_STATE]->(s) CREATE (s)-[:ESTABLISHED_IN]->(e)
  WITH s MATCH (ck:Chunk {chapter:6, index: <근거 청크>}) CREATE (s)-[:EVIDENCED_BY]->(ck)
  ```
  (index 값은 원문 대조로 실행 시 확정. `__Entity__` 라벨은 붙이지 않아 resolver 대상 밖 — 수동 노드는 병합 불필요)
- #3은 판단여지 수준이므로 재발해도 수용 가능(패치 선택).

## 실행 순서

1. **extractor.py / extraction_examples.py 보강** (a)~(e) → 검증: 모듈 import + few-shot JSON 파싱
2. **1~6화 재인덱싱** — 이미 예정된 PART_OF 반영과 **한 번에**(DB 초기화 후 순차, 비용 ~$0.4대).
   `produceSelfRel:false`·PART_OF도 이 재인덱싱에서 함께 반영됨
3. **표적 검증** (아래) → 4. **재발분만 수동 패치** → 재검증

## 검증

**표적 4건** (재인덱싱 후 해당 노드 직접 확인):
1. Event 5.2 description — 김독자가 자신·유상아의 **죽음**을 예상하는 방향으로 서술됐는가
2. Event 4.2 description — 이현성이 이름을 밝힌 것으로 읽히지 않는가(계급·부대 발화와 김독자 인지 구분)
3. Event 5.4 description — 폭발 원인이 "밝혀지지 않는다" 단정 대신 암시 수준으로 서술됐는가
4. 김독자 ch6 상태에 '등장인물 일람' 보유 State가 존재하는가
5. Organization '성좌' 노드가 존재하는가(+ 김남운의 '두 성좌 관심' 상태가 ABOUT으로 이어졌으면 더 좋음)
```cypher
MATCH (e:Event) WHERE e.chapter IN [4,5] AND e.story_order IN [4.2, 5.2, 5.4]
RETURN e.story_order, e.name, e.description;
MATCH (c:Character {name:'김독자'})-[:HAS_STATE]->(s)-[:ESTABLISHED_IN]->(e:Event {chapter:6})
RETURN s.name, s.description;
MATCH (o:Organization) WHERE o.name CONTAINS '성좌' RETURN o.name, o.description;
```

**회귀 확인** (프롬프트 보강이 추출 범위를 흔들지 않았는지):
- 라벨별 노드 수를 이번 실측 기준선(Character 8 / Event 43 / CharacterState 60 / Item 2 / Org 6 / Location 7)과
  대조 — PART_OF 신설·#4 상태 추가 외 큰 변동이 없어야 함
- 기존 검증 셋 재실행: 근거 없는 Event/CharacterState 0, 배열 description 0, self-loop 0(이번엔 예방 적용됨),
  정준-이름 라벨 중복 name 0
- **PART_OF 생성 확인**: `인사팀→대기업 계열사`, `재무팀→대기업 계열사` 등
  ```cypher
  MATCH (a:Organization)-[:PART_OF]->(b:Organization) RETURN a.name, b.name;
  ```

## 스코프 밖 (명시)

감사의 경계·판단여지 항목은 **의도적으로 제외** — 할머니(무명, 스키마의 단역 제외 원칙상 정당),
국무총리 Character(무명 직위), 이현성 원작 괴리 인지(내적 독백), 태풍여고 교실 LOCATED_IN·한명오
APPEARS_IN(경미, 재인덱싱에서 자연 개선되면 수용). 이들은 실측 후 필요 시 다음 라운드.
(성좌는 사용자 결정으로 스코프에 **포함**됨 — #5)
