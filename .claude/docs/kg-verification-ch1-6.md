## 🔴 S1 — 치명적 (조회 결과가 틀림)

### 1. `3807칸`/`3907칸` 오병합 — 사건 17건의 무대가 전부 틀림

원작에는 **세 칸**이 명확히 구분된다.

| 객차 | 역할 |
|---|---|
| 3707칸 | 앞 칸. 프롤로그 주인공이 탑승, 5화에서 폭발 발생 |
| **3807칸** | **김독자·유상아 탑승. 2~6화의 실제 무대** |
| 3907칸 | 뒷칸. 4화에서 최초 살인이 발생 |

그러나 그래프에는 `Location:지하철 3907칸` **하나만** 있고, ch2~6 사건 **17건 전부**가 이 노드에 `HOSTS`로 매달려 있다. 3807칸·3707칸 노드는 아예 없다.

**원문 근거**
- `"나는 반사적으로 내가 탄 열차 칸의 번호를 확인했다. / [3807]."`
- `"내가 지금 타고 있는 칸은, 프롤로그의 주인공이 타고 있던 칸의 바로 '뒤 칸'이었다."`
- `"3807칸의 앞쪽 문이 벌컥 열리며 전기가 돌아온 것은 그때였다."` (3화)
- `"통로 쪽 창문으로 피바다가 된 3907 칸의 정경이 비치고 있었다."` (4화) — 김독자는 3907칸에 **들어갈 수조차 없다**

**그래프 자체가 모순** (원문 없이도 검출 가능)
- Event 5.1은 title이 `"3807칸 할머니 집단 폭행"`인데 `HOSTS`는 3907칸
- Event 3.0은 description이 `"지하철 3807칸 앞쪽 문이 열리고"`인데 3907칸에 연결
- 노드 description에 두 칸의 설명이 혼재: `"살인 사건이 발생한 지하철 뒷칸"`(3907) + `"김독자와 유상아가 퇴근길에 이용한 전동차"`(3807)

**원인**: `CombiningFuzzyResolver`(RapidFuzz WRatio)가 한 글자 차이(3807↔3907)를 표기 변형으로 오판해 병합. **숫자를 포함한 고유명에 fuzzy 병합을 적용하면 위험**하다는 구조적 교훈.

**영향**: "김독자가 어느 칸에 있었는가", "살인이 어디서 일어났는가" 같은 질의가 전부 틀린 답을 낸다. 장소 기반 모순 탐지가 무력화된다.

### 2. 6화 `story_order` 역전 — 인과가 뒤집힘

| 그래프 | 원문 위치 | 실제 순서 |
|---|---|---|
| 6.1 메뚜기알 살해·체력 Lv.10 | 316~318행 | **나중** |
| 6.2 맥가이버 칼 공격·어깨 상처 | 254행 | **먼저** |

원문의 인과는 다음과 같다.

```
어깨 상처("가까스로 심장을 빗나간 칼날이 어깨를 깊이 긋고 지나갔다", 254행)
  → 코인으로 체력 강화([2700코인을 '체력'에 투자합니다] / [체력 Lv.1 -> 체력 Lv.10], 316~318행)
  → 그 후엔 칼이 통하지 않음("마치 단단한 바위를 긁은 것처럼 생채기만 남은 피부", 328행)
```

순서가 뒤집히면 "체력 강화 후에 상처를 입음"이 되어 **인과가 붕괴**한다. 연동된 `CharacterState`(어깨/칼날 상처 so=6.2, 체력/Lv.10 so=6.1)도 같이 잘못 배치됐다.

## 🟠 S2 — 구조적 오류

### 3. `LOCATED_IN` 자기참조 2건

```
지하철 3907칸           -[LOCATED_IN]-> 지하철 3907칸           (self_loop = TRUE)
태풍여고 2학년 B반 교실  -[LOCATED_IN]-> 태풍여고 2학년 B반 교실  (self_loop = TRUE)
```

`LOCATED_IN`은 "한 단계 위 장소"를 가리켜야 하는데 자기 자신을 가리킨다. 계층 순회 시 무한 루프 위험. 1번·4번 병합의 직접적 부산물이다.

### 4. `태풍여고`(학교) + `2학년 B반 교실` 오병합

교실 노드의 description에 `"이지혜가 재학 중인 여학교"`가 섞여 있다. 상위 장소(학교)와 하위 장소(교실)가 한 노드로 뭉개져 `교실 -[LOCATED_IN]-> 태풍여고` 계층이 소멸했다.

부수 문제: **이지혜의 태풍여고 소속 상태가 없다**(생사=생존만 존재). 김남운은 `소속`+`ABOUT`→청일고교가 정상 생성돼 있어 처리가 비일관적이다.

### 5. `INVOLVED_WITH role='사용자'` — 스키마 위반 + 소유 궤적 파괴

```
김독자 -[INVOLVED_WITH role='사용자']-> 곤충 채집망
김남운 -[INVOLVED_WITH role='사용자']-> 휴대용 맥가이버 칼
```

스키마가 규정한 role은 **저자/독자/제작자**뿐이며, 소지·사용은 `CharacterState(attribute='소유')` + `ABOUT`→Item으로 표현해야 한다.

- **맥가이버 칼**: 소유 상태(보유 6.2 / 상실 6.3)가 이미 존재 → **중복이자 규칙 이탈**
- **곤충 채집망**: 이 잘못된 관계가 **소유 상태를 대체해 버려** 5화의 획득→상실 궤적이 통째로 소실
  - `"나는 아이에게서 채집망을 빼앗아 들었다."` (획득 = 보유)
  - `"다른 한 손으로 쥐고 있던 채집망을 힘껏 던졌다."` (상실)
  - → 6화 서두(곤충 흩뿌림)와의 인과가 그래프상 끊긴다

### 6. 회차 요약(`Chapter.summary`) 환각 2건 — 오염이 다음 회차로 전파

| 회차 | 요약 내용 | 원문 사실 |
|---|---|---|
| 1화 | `"tls123와 독자-작가 이상의 특별한 관계를 드러낸다"` | 정반대. `"감탄은 졸렬한 질투심으로 바뀌었다"`, `"이제 성공했다고 답장도 안 하고 가버리나?"` |
| 2화 | `"웹소설 『멸망 이후의 세카이』 작가에게 ... 메일을 받지만"` | 『멸망 이후의 세카이』(작가 싱샹숑)는 김독자가 흘끗 본 **다른 작품**. 메일 발신인·유료화 대상은 **멸살법(tls123)** |

요약은 다음 회차의 `novel_context`로 주입되므로 **오류가 누적 전파**된다.

## ⚠️ 검증 과정의 오탐 (정정)

병렬 검증 에이전트 4개가 **"Organization/Item 노드의 라벨이 `__KGBuilder__`로 잘못 부여됨"**을 보고했으나, 이는 **덤프 쿼리가 `labels(n)[0]`로 첫 번째 라벨만 출력한 표시 오류**였다. 실제 라벨을 재확인한 결과 정상이다.

```
["Item"],         "곤충 채집망" / "휴대용 맥가이버 칼" / "tls123가 보낸 선물" / "멸망한 세계에서..."
["Organization"], "6502부대" / "대기업 계열사" / "재무팀" / "청일고교"
```

→ **실제 데이터 문제 아님. 조치 불필요.**

## 우선 조치 제안

1. **resolver에 숫자 가드 추가** — 고유명에 포함된 숫자가 다르면 병합 금지.
   S1-1이 근본 원인이며, 이것만 고치면 HOSTS 17건·Location description·LOCATED_IN self-loop·4화 요약이 **연쇄적으로** 정리된다. 가장 비용 대비 효과가 크다.
2. **`INVOLVED_WITH.role` 열거 제한** — 프롬프트에서 role을 저자/독자/제작자로 못 박고, 그 외 소지·사용은 반드시 `CharacterState(소유)`+`ABOUT`으로 표현하도록 강제.
3. **시스템 메시지 상태화 규칙 추가** — `[대괄호]` 시스템 메시지로 확정되는 수치(코인·업적·능력치·스킬)는 `CharacterState`로 승격하도록 프롬프트에 명시.
4. **요약 프롬프트에 환각 방지 지시** — "원문에 없는 고유명·관계를 만들지 말 것". 1·2화 환각이 `novel_context`로 전파되는 경로를 차단.
5. **story_order 검증 규칙** — 같은 회차 내 사건 순서가 원문 등장 순서와 일치하는지 확인하는 지시 강화(S1-2).
